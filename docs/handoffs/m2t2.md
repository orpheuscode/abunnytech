## What Stage 4 can rely on

### `DistributionRecord` (one per platform post attempt)

Written to SQLite at `./db/stage3.db`, table `distribution_records`.

Indexed columns for fast Stage 4 queries:

| Column | Type | Notes |
|---|---|---|
| `record_id` | TEXT PK | UUID |
| `package_id` | TEXT | maps back to Stage 2 ContentPackage |
| `identity_id` | TEXT | maps back to Stage 0 IdentityMatrix |
| `platform` | TEXT | `tiktok \| instagram \| youtube` |
| `status` | TEXT | `posted \| failed \| skipped \| dry_run` |
| `dry_run` | INT | 1 = dry-run record, 0 = live |
| `posted_at` | TEXT ISO | null on failure/dry-run |

Full JSON blob in `data` column. Use `Stage3Store.list_distribution_records()` to query.

**Key status values Stage 4 should handle:**
- `posted` — successful live post; `post_id` and `post_url` are set
- `dry_run` — simulated post; no real platform IDs
- `failed` — error in `record.error`

### `DMConversationRecord` (one per comment-triggered DM flow)

Table `dm_conversations`.

| Column | Type | Notes |
|---|---|---|
| `conv_id` | TEXT PK | UUID |
| `platform` | TEXT | platform of the originating comment |
| `user_id` | TEXT | commenter's platform user ID |
| `fsm_state` | TEXT | current FSM state (see below) |
| `dry_run` | INT | |
| `updated_at` | TEXT ISO | advances on every state transition |

**FSM states Stage 4 can observe:**
- `reply_planned` — public reply drafted, awaiting execution
- `dm_planned` — DM drafted (confirmed follower path), awaiting execution
- `sent` — reply or DM was sent; `reply_id` / `dm_message_id` populated
- `converted` — conversion event recorded; `conversion_events` list populated
- `rejected` — spam detected, no action taken

**Conversion event types:** `link_clicked | product_viewed | purchase | follow | dm_reply`

## Integration points Stage 4 needs to build

1. **Analytics correlation:** Join `DistributionRecord.record_id` → platform analytics by `post_id` to populate `PerformanceMetricRecord`.

2. **Conversion attribution:** Read `DMConversationRecord.conversion_events` to attribute revenue/follows back to specific posts.

3. **OptimizationDirective feedback loop:** When Stage 4 emits directives, Stage 3 can re-prioritize its scheduler queue and adjust `PlatformTarget.max_posts_per_day`.

4. **RedoQueueItem:** If Stage 4 determines a post underperformed, it can enqueue a new `ContentPackage` → Stage 3 re-runs the posting cycle.

## Compliance notes

- All posts include `ai_disclosure_footer` in caption — non-negotiable, enforced in executor.
- All DMs set `ai_disclosure=True` — enforced in executor.
- No mass-DM patterns: DMs are only triggered by explicit comment keywords, one-to-one.
- Rate limits enforced at scheduler level (max posts/day per platform).
- Dry-run mode simulates full flow without any real platform interactions.
