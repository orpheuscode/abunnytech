# m3t4 Handoff — Pipeline Orchestration

## Overview

The orchestrator is the top-level control plane that wires stages 0–4 together
without duplicating their internals.  It lives in `apps/orchestrator/` and
exposes both a **Typer CLI** and an importable `PipelineRunner` class.

## Architecture

```
packages/config/          Central PipelineConfig (wraps packages/shared/config)
apps/orchestrator/
  context.py              RunContext — mutable state threaded through a run
  pipeline.py             PipelineRunner — stage sequencing and error handling
  cli.py                  Typer CLI entry point
scripts/runbooks/
  demo.py                 One-command hackathon demo
```

## Pipeline Sequencing

Each stage produces outputs that the next stage consumes via `RunContext`:

```
Stage 0  (Identity)     → identity_id
Stage 1  (Discover)     → trending items (stored in DB, count in artifacts)
Stage 2  (Generate)     → blueprint_id, content_package_id
Stage 3  (Distribute)   → distribution_record_id
Stage 4  (Analyze)      → metrics + optimization directives
```

The **demo** runs stages 0 → 4 in sequence, stopping on first error.

The **feedback loop** takes an existing `identity_id` +
`distribution_record_id`, runs stage 4, then re-runs stages 1 → 3 with the
refreshed context.

## Environment Variables

| Variable | Default | Purpose |
|---|---|---|
| `DRY_RUN` | `true` | When true, stage 3 skips real posts |
| `KILL_SWITCH` | `false` | Emergency halt — pipeline refuses to start |
| `FEATURE_STAGE5_MONETIZE` | `false` | Gate for the monetization stage |
| `LOG_LEVEL` | `INFO` | structlog filter level (`DEBUG`, `INFO`, `WARNING`, `ERROR`) |
| `DATABASE_URL` | `sqlite+aiosqlite:///./abunnytech.db` | Async SQLAlchemy connection string |
| `OPENAI_API_KEY` | (empty) | Used by real adapters when dry-run is off |
| `ELEVENLABS_API_KEY` | (empty) | Used by real voice adapters |

Copy `.env.example` → `.env` and fill in any secrets before running.

## CLI Commands

All commands are invoked via `python -m apps.orchestrator.cli <command>`.

| Command | Description |
|---|---|
| `run-demo` | Full happy-path: stage 0 → 4 with demo defaults |
| `run-stage0` | Create default identity |
| `run-stage1 --identity-id ID` | Discover trending |
| `run-stage2 --identity-id ID --topic T` | Generate blueprint + render |
| `run-stage3 --content-package-id ID` | Distribute (respects `DRY_RUN`) |
| `run-stage4 --identity-id ID --distribution-record-id ID` | Analyze + optimize |
| `run-feedback-loop --identity-id ID --distribution-record-id ID` | Analyze → re-run 1–3 |
| `status` | Print resolved config and feature flags |

## One-Command Demo

```bash
python -m scripts.runbooks.demo
```

Seeds a default identity, discovers trends, generates content, distributes
(dry-run), and collects analytics.  Output includes all IDs needed to run a
follow-up feedback loop.

## Integration Notes

- **For dashboard / integration agents**: import `PipelineRunner` and
  `RunContext` directly — they are pure async Python with no CLI dependency.
- **For the HTTP control plane**: mount stage routers from `stages/*/router.py`
  into a FastAPI app and call `PipelineRunner` from endpoint handlers.
- **State**: all pipeline records are in a single SQLite database via
  `packages.shared.db`.  Stages write `PipelineRecord` rows keyed by
  `contract_type` and `stage` — query those directly for dashboards.
- **Kill switch**: set `KILL_SWITCH=true` to immediately halt any pipeline
  invocation.  The check happens at pre-flight and before each stage.

## Extending

To add a new stage to the pipeline:

1. Add its canonical name to `STAGE_ORDER` in `packages/config/loader.py`.
2. Add a `run_stageN` method in `apps/orchestrator/pipeline.py` that
   instantiates the stage service and feeds/reads `RunContext`.
3. Wire it into `run_demo`'s `stage_runners` list.
4. Add a CLI command in `apps/orchestrator/cli.py`.
