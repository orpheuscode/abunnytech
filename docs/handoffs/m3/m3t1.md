# m3t1 Handoff – State Layer & API

## Architecture Overview

```
packages/state/          ← storage layer (no web framework dependency)
├── base.py              Repository[T] abstract interface
├── models.py            Pydantic v2 models for all 12 entity types
├── sqlite.py            Database + SQLiteRepository implementation
├── registry.py          RepositoryRegistry (typed access to all 12 repos)
├── events.py            EventBus + JobRegistry for orchestration
└── fixtures.py          Demo seed data + seed_all() helper

apps/api/                ← FastAPI thin CRUD surface
└── main.py              Lifespan, router factory, all endpoints
```

## SQLite Storage Pattern

Each table uses a **JSON-document-store** layout:

```sql
CREATE TABLE IF NOT EXISTS [table_name] (
    id TEXT PRIMARY KEY,
    data TEXT NOT NULL,      -- full Pydantic model as JSON
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);
```

This means:
- **No DDL migrations needed** when model fields change – just add fields with defaults.
- The `data` column holds the complete serialized model (including `audit_log`).
- `id`, `created_at`, `updated_at` are denormalized into columns for indexing/ordering.
- SQLite WAL mode is enabled for concurrent reads.

### Migrating to Postgres / Supabase

1. Implement a new class that satisfies `Repository[T]` (see `packages/state/base.py`).
2. Replace `SQLiteRepository` in `RepositoryRegistry.__init__` with your new implementation.
3. If using Postgres JSONB, the same document pattern works out of the box.
4. If using normalized tables, map each model's fields to columns in your implementation.

## API Quick Reference

All entity endpoints follow the same pattern under `/api/{collection}`:

- `GET    /api/{collection}?limit=100&offset=0` – paginated list
- `GET    /api/{collection}/count` – item count
- `GET    /api/{collection}/{id}` – single item
- `POST   /api/{collection}` – create (body = full model JSON)
- `PUT    /api/{collection}/{id}` – full replace
- `DELETE /api/{collection}/{id}` – delete

Utility endpoints:

- `GET  /health` – returns `{"status": "ok"}`
- `GET  /api/collections` – list all collection names
- `POST /api/seed` – insert demo fixture data

## Event Bus & Job Registry

`packages/state/events.py` provides two orchestration primitives:

```python
from packages.state.events import EventBus, JobRegistry

bus = EventBus()
bus.on("content.created", my_handler)
await bus.emit("content.created", package_id=uuid)

jobs = JobRegistry()
jobs.register("generate_thumbnail", thumbnail_handler)
await jobs.run("generate_thumbnail", video_url="...")
```

These are in-process only. For cross-process orchestration, wrap them with
a message queue (Redis pub/sub, etc.) in a future iteration.

## Integration Notes for Other Terminals

**Importing models**: All 12 models are available from `packages.state.models`
or from the top-level `packages.state` namespace.

**Getting a repository**: Build a `Database` + `RepositoryRegistry`, or use the
running API via HTTP.

```python
from packages.state.sqlite import Database
from packages.state.registry import RepositoryRegistry

db = Database("abunnytech.db")
await db.connect()
registry = RepositoryRegistry(db)
item = await registry.identity_matrix.get(some_uuid)
```

**Contract compatibility**: The models in `packages.state.models` mirror the
shapes declared in `packages/contracts`. When the contracts owner publishes
the remaining modules (discovery, content, distribution, analytics,
monetization), downstream code can switch imports with no schema changes.
