# Handoff: M1T4 Stage 2 → Stage 3 (distribution)

## Official handoff object

Stage 3 should treat **`ContentPackage`** (from `pipeline_contracts.models.content`) as the only canonical publishable bundle per deliverable. The primary short for a run is stored as the repository artifact **`content_package`** (JSON). Carousel and story placeholders are stored under **`stage2_variant_packages`** as:

```json
{ "packages": [ /* ContentPackage, ... */ ] }
```

Each entry is a full `ContentPackage` (including `package_id`, `primary_video` with `mime_type` either `video/mp4` or `image/png` for static slides).

## Assumptions Stage 3 can rely on

- **`primary_video.path`** points at a filesystem path (or future object key) relative to the worker that ran Stage 2; in **dry-run** it is a placeholder file under `artifacts/<run_id>/stage2/primary_demo.mp4`.
- **`caption`** and **`hashtags`** are ready for posting; **`disclosure_tag`** may combine the demo tag with `IdentityMatrix.persona.disclosure_line` when `PIPELINE_DISCLOSURE_DEMO` is enabled.
- **`matrix_id`** and **`blueprint_id`** correlate back to Stage 0 / Stage 1 artifacts for audit and re-runs.

## Artifacts Stage 2 writes (for ops / mux, not required for minimal post)

Under `artifacts/<run_id>/stage2/`:

| File | Purpose |
|------|---------|
| `elevenlabs_audio_request.json` | ElevenLabs-compatible TTS body (adapter may remap field names). |
| `nano_banana_video_request.json` | Video generation request stub. |
| `post_production_manifest.json` | Overlays, subtitles path, 9:16 crop, trending underlay. |
| `adapted_script.json` | Timed script segments (hook / beats / CTA). |
| `caption_metadata.json` | Caption + hashtag + title + alt text snapshot. |
| `placeholder_asset_manifest.json` | Paths to raw video, voiceover, primary output, subtitles. |
| `subtitles.srt` | Minimal stub SRT for dry-run subtitle path resolution. |

Stage 3 does **not** need to parse these unless it performs server-side mux or QC.

## Redo / optimization flow (Stage 4 → Stage 2)

- **`redo_queue`** artifacts (list of `RedoQueueItem`) may patch the blueprint before generation. The first item whose `blueprint_id` matches (or is omitted) is applied.
- **`optimization_directives`** entries with `target_stages` including `stage2` apply `envelope.payload` using the same key conventions as redo payloads.

Stage 3 should not mutate redo queues; it may append distribution outcomes that Stage 4 turns into new redo items.

## Downstream needs for Stage 3

1. **Scheduling**: iterate `stage2_variant_packages.packages` when platforms require carousels or story sequences; otherwise use `content_package` only.
2. **MIME handling**: skip video upload for `image/png` rows; use carousel APIs where available.
3. **Trending audio**: when present, `VideoBlueprint.audio_id` flowed into the post-production manifest underlay spec; actual platform “use this sound” IDs may still need a Stage 3 mapping table.
4. **Idempotency**: `package_id` is unique per generated package; replays should use new IDs from a fresh Stage 2 run.

## Out of scope for Stage 2 (M1T4)

- Editing `stage-0-1-2/packages/contracts`, Stage 3, API routes, or browser runtime (per milestone boundaries).
- Guaranteed valid MP4 bytes in dry-run mode (placeholders only).
