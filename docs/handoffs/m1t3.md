# Handoff: Stage 1 discovery → Stage 2 generation (m1t3)

## What Stage 1 produces

After a successful offline or live run, Stage 1 emits a JSON-serializable bundle (`Stage1Artifacts`) with four top-level keys:

1. **`video_blueprints`** — list of objects validating as `pipeline_contracts.models.VideoBlueprint`. Each item is an independent creative brief: hook, outline beats, caption draft, hashtags, optional `audio_id`, and `duration_seconds_target`. Stage 2 should treat `blueprint_id` and `matrix_id` as stable foreign keys into the run store.

2. **`trending_audio`** — list of `TrendingAudioItem` dicts (`audio_id`, `title`, `platform`, `trend_score`, optional `bpm`, `source_label`). Stage 2 can map `VideoBlueprint.audio_id` back to metadata for licensing prompts or render notes.

3. **`competitor_watchlist`** — list of `CompetitorWatchItem` dicts for ongoing research; Stage 2 does not need them to render, but they explain hook patterns and positioning used during discovery.

4. **`ranked_queue`** — ordered list of `{ rank, blueprint_id, copyability_score, content_tier, candidate_id, score_breakdown }`. Stage 2 should consume blueprints in `rank` order when batching work: lower rank means higher copyability / safer remix priority. `score_breakdown` is a float map for debugging (e.g. transcript weight, brand penalty).

`discovery_plan` is included as an optional dict for audit (seed queries, platforms, tier targets).

## How Stage 2 should load inputs

- Resolve the active run’s `identity_matrix` (Stage 0) and the chosen `video_blueprint` (or iterate `ranked_queue` and load each `blueprint_id`).
- Pass `VideoBlueprint` plus identity context into generation adapters (script, avatar, voice, render). Do not depend on `candidate_id` unless you add a Stage 2 extension; it is for traceability to source shorts.
- If `audio_id` is set, join against `trending_audio` for human-readable titles or BPM hints in prompts.

## Offline demo

From the repo root (after `uv sync --group dev`):

```bash
uv run abunny-stage1-dry-run --json
```

Or call `run_stage1_fixture_pipeline` / `load_identity_fixture` from `abunny_stage1_discovery` using `examples/stage1/fixtures`.

## Browser and ML boundaries

- Stage 1 agent code defines **Protocols** for browser/runtime and for analysis vendors (`browser_ports.py`, `analysis_ports.py`). A future `packages/browser_runtime` implementation should satisfy the discovery/metadata/download planner interfaces without changing Stage 1’s orchestration.
- TwelveLabs- and Gemini-shaped **mocks** live under `adapters/vendor_shim.py` and delegate to fixture-backed analysis for local runs.
